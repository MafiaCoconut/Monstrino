# Kubernetes Cluster Architecture

## Overview

The Monstrino platform runs on a Kubernetes cluster hosted in a homelab
environment.

All services are deployed as Kubernetes **Deployments** and exposed
internally through **Services** and externally via **Ingress**.

## Goals

-   container orchestration
-   service isolation
-   horizontal scalability
-   reproducible deployments

## Deployment Model

Services are deployed as:

Deployment → Pod → Container

Each service runs independently and can be scaled without affecting
other components.

## Networking

Internal communication occurs through Kubernetes Services using cluster
DNS.

Example:

service-name.namespace.svc.cluster.local

# Environment Strategy

## Environments

The system currently supports three runtime environments:

-   local
-   test
-   production

## Local

Used for development and experimentation.

Characteristics:

-   reduced infrastructure
-   local datasets
-   developer tools enabled

## Test

Used for validating new deployments and integration testing.

Characteristics:

-   realistic configuration
-   isolated namespace
-   staging datasets

## Production

Runs the public platform and collectors.

Characteristics:

-   production data
-   stable configuration
-   restricted access

# Kubernetes Namespace Structure

## Purpose

Namespaces are used to isolate environments and workloads inside the
Kubernetes cluster.

## Current Structure

Typical namespaces include:

-   local
-   test
-   production

## Benefits

-   logical separation of services
-   easier resource management
-   environment isolation

## Usage

All services belonging to the same environment are deployed in the same
namespace.

# Storage Architecture

## Media Storage

Media assets such as images are stored in object storage.

The platform uses **MinIO**, which provides an S3-compatible API.

## Reasons for Choosing MinIO

-   self-hosted infrastructure
-   S3 compatibility
-   good performance for large files

## Storage Structure

Typical object structure:

bucket / images / `<hash or identifier>`{=html}

This allows deterministic storage and retrieval of assets.

# Ingress and Networking

## Ingress

External access to services is handled through an ingress controller.

The ingress layer routes HTTP requests to the appropriate backend
service.

## Responsibilities

-   routing
-   TLS termination
-   domain management

## Request Flow

Client → Ingress → Service → Pod

## Internal Networking

Services communicate using Kubernetes service discovery rather than
external URLs.

# Observability Stack

## Monitoring

The platform uses a monitoring stack for observing system health.

Typical components include:

-   Prometheus for metrics collection
-   Grafana for visualization

## Metrics

Important metrics:

-   service latency
-   error rates
-   resource usage

## Purpose

Observability ensures that ingestion pipelines and APIs remain reliable
and detectable when failures occur.
