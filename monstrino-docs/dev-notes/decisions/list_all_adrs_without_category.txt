MONSTRINO — COMPLETE ADR LIST

ADR-01
Title: Focus the product domain on Monster High instead of LEGO or Funko

Context:
Изначально проект планировался как каталог LEGO. После анализа конкурентов стало ясно:
- LEGO каталоги уже существуют в большом количестве
- Funko POP каталоги также сильно насыщены
- Monster High сайты в основном поддерживаются вручную

Decision:
Сфокусировать продукт на Monster High ecosystem.

Consequences:
+ менее конкурентная ниша
+ автоматизация данных имеет ценность
+ более уникальный продукт


ADR-02
Title: Make automated data acquisition the core product capability

Context:
Большинство сайтов каталогов обновляются вручную.

Decision:
Monstrino должен автоматически:
- находить новые релизы
- собирать данные
- генерировать страницы

Consequences:
+ высокая масштабируемость
+ минимальный ручной труд
- сложная архитектура ingestion


ADR-03
Title: Prioritize release archive over user features for MVP

Context:
Ранее начали разрабатываться:
- JWT auth
- пользовательские коллекции
- user pages

Но без каталога эти функции бесполезны.

Decision:
Первый релиз Monstrino должен включать только архив релизов.

Consequences:
+ быстрый MVP
+ фокус на core functionality


ADR-04
Title: Separate ingestion data from canonical catalog data

Context:
Данные источников могут:
- содержать ошибки
- иметь разный формат
- быть неполными

Decision:
Использовать две зоны данных:

parsed tables
parsed_releases
parsed_series
parsed_characters
parsed_pets

canonical tables
releases
series
characters
pets

Парсер пишет только в parsed tables.
catalog-importer переносит данные в canonical tables.

Consequences:
+ безопасная обработка данных
+ возможность повторного импорта


ADR-05
Title: Use database processing_state for ingestion workflows

Context:
Kafka pipeline создавал сложности:
- ошибки блокируют очередь
- сложно отслеживать ошибки

Decision:
Использовать поле состояния записи:

processing_state
init
processing
processed
error

Consequences:
+ простой retry
+ удобная диагностика


ADR-06
Title: Use external_id as main ingestion identifier

Context:
URL может:
- изменяться
- содержать параметры

Decision:
Использовать идентификатор:

source + external_id

Consequences:
+ стабильный идентификатор
+ удобный reparse
