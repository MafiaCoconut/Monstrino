# Platform Architecture Evolution

## Current State

Monstrino currently runs as a set of independent microservices deployed
in a single Kubernetes cluster in a homelab environment.

Key capabilities:

-   data ingestion pipelines
-   media processing
-   market data collection
-   AI enrichment
-   API access for frontend applications

## Mid-Term Goals

The next architectural phase focuses on:

-   stabilizing ingestion pipelines
-   expanding market data coverage
-   improving AI-assisted data processing
-   strengthening observability and monitoring

## Long-Term Goals

Future architectural evolution may include:

-   separating production and testing clusters
-   introducing distributed storage replication
-   improving global performance via CDN
-   scaling ingestion pipelines horizontally

# Data Collection Strategy

## Objective

The goal of the platform is to build the most complete dataset possible
for collectible releases.

Data must be collected as early as possible because historical data
cannot always be reconstructed.

## Key Sources

Important data sources include:

-   collector websites
-   retailer pages
-   secondary market platforms
-   community databases

## Strategy

The strategy focuses on:

1.  continuous discovery of new releases
2.  regular price collection from market sources
3.  historical tracking of price evolution

This ensures the platform accumulates long-term datasets that increase
in value over time.

# AI Integration Roadmap

## Phase 1 --- Basic AI Assistance

Initial AI integration focuses on:

-   text normalization
-   entity extraction
-   metadata cleanup

This reduces manual data cleaning and improves catalog consistency.

## Phase 2 --- Advanced Extraction

Future models will assist with:

-   character recognition from images
-   automatic classification of releases
-   improved parsing of unstructured descriptions

## Phase 3 --- Intelligent Automation

Long-term goals include:

-   AI-driven ingestion validation
-   automatic anomaly detection in data
-   smarter data enrichment pipelines

# Scaling Strategy

## Current Infrastructure

The system currently runs on a single Kubernetes cluster hosted on one
server.

Namespaces separate environments such as:

-   local
-   test
-   production

## Scaling Plan

When traffic or workloads increase:

-   a dedicated production server will host the production cluster
-   additional nodes may be introduced
-   data replication will be implemented

## Reliability Improvements

Future reliability improvements may include:

-   redundant storage
-   backup clusters
-   automated failover mechanisms
